How it's built

OpenMRS as the core, custom modules around the edges

Rather than writing a new EHR, NCDMS configures OpenMRS 3 as the clinical backbone — patients, encounters, observations, the concept dictionary — and adds small companion modules for the operational parts of a dialysis unit that don't fit naturally into a general EHR.

Core platform

OpenMRS 3 (O3 Reference Application)

The system runs on the OpenMRS 3 Reference Application, deployed via Docker Compose. OpenMRS provides the data model that everything else builds on: a Concept Dictionary for clinical terms, and a Visit → Encounter → Observation hierarchy for recording care over time. Nephrology-specific data — serum creatinine, eGFR, dialysis details, vascular access type — is captured through custom clinical forms built on this foundation, rather than through generic visit notes.

  • OpenMRS 3 (O3 Reference Application)
  • Docker Compose
  • Concept Dictionary (CIEL via OCL)
  • FHIR2 module
  • Custom O3 form schemas (JSON)

This separation keeps the clinical record itself close to OpenMRS conventions — which matters for long-term maintainability and interoperability — while letting the operational tools evolve faster and more specifically to this unit's needs.

Standards

Built on existing health informatics standards where possible

Terminology

Clinical concepts are sourced from CIEL via OpenConceptLab where available, with LOINC used for laboratory observations and ICD-10 for diagnosis coding aligned to Ministry of Health reporting requirements.

Interoperability

The FHIR2 module is kept enabled, so the underlying clinical data remains exportable in a standard format — important for a system that may eventually need to exchange data with other hospital systems or national health information infrastructure.
